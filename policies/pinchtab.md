# Pinchtab — Policy

## Назначение

Локальный HTTP-мост к реальному Chrome. Главная цель: read-first.
Действия (click/type) — вторично и строго ограниченно.

## Жёсткие запреты (без явного разрешения пользователя)

Запрещены любые действия (и часто даже чтение) на:
- Банки/платежи/крипто/биржи, любые финансовые кабинеты
- Почта (Gmail/Outlook/Webmail), мессенджеры, соцсети с восстановлением паролей
- Админки/панели управления (хостинг, VPS-панель, домены/DNS, CDN, SSO, менеджеры паролей)
- Страницы с покупкой/оплатой/подтверждением транзакции
- Страницы с персональными данными (документы, счета, медданные)
- URL с секретами: token=, key=, signature=, auth=, session=, magic-links, reset links

Если пользователь явно просит запрещённое:
1. Предупредить о риске
2. Предложить безопасную альтернативу (API/SMTP/ручные шаги)
3. Делать только после явного подтверждения

## Мини-чеклист перед navigate(url)

- URL публичный http/https
- Не содержит токенов/секретов в query
- Не относится к запрещённым категориям
- Не является localhost/приватной подсетью/сырым IP

Если сомнения — не идти, запросить безопасную альтернативу.

## Безопасность инфраструктуры

- Pinchtab только на localhost, BRIDGE_TOKEN обязателен
- Пустой изолированный Chrome-профиль, без паролей/синхронизации
- Токен не логировать, не показывать пользователю

## Лестница чтения (главное правило)

Всегда от дешёвого к дорогому:

1. `browser.sh navigate <url>` → дождаться загрузки
2. `browser.sh text` → дешёвый readable text
3. Если текста достаточно → анализировать/отвечать ✅
4. Если текст пустой/обрезанный → `browser.sh snapshot_interactive`
5. Если нужен полный контекст → `browser.sh snapshot_full` (дорого, только когда иначе нельзя)

## Разрешённые действия (read-adjacent)

- Закрытие cookie-баннеров
- "Read more" / "Expand" / "Show more"
- Переход по ссылкам внутри статьи
- Пагинация (Next / Page 2)
- Открытие печатной версии/режима чтения

## Запрещённые действия

- Submit/Send/Pay/Confirm/Buy/Delete/Publish
- Отправка форм, публикация, оплата, удаление
- Любые необратимые действия

## E-mail

По умолчанию Pinchtab не для webmail. Если нужен авто-email:
предпочтительно SMTP/API. Webmail — только с отдельным аккаунтом и явным разрешением.

## ⚠️ Управление жизненным циклом — ОБЯЗАТЕЛЬНО

### Проблема: Chrome восстанавливает сессию

При перезапуске `run.sh restart` или `run.sh start` — Chrome **автоматически восстанавливает все открытые вкладки** прошлой сессии. Тяжёлые страницы (SEMrush, SPA) перегружаются заново и начинают жрать CPU. Инцидент 2026-02-23: зависший renderer от SEMrush-сессии накопил 8+ часов CPU-времени → load average 20+ на 2-ядерном сервере.

### Правило: завершение работы с Pinchtab

**Всегда перед `run.sh stop`:**
1. Закрыть все рабочие вкладки — навигировать на пустую страницу:
   `browser.sh navigate "about:blank"` — для каждой тяжёлой вкладки
2. Проверить что осталось: `browser.sh tabs`
3. Только потом: `~/.pinchtab/run.sh stop`

**Если работа была с SEMrush** — обязательно. SEMrush-страницы особенно тяжёлые при восстановлении.

### Диагностика нагрузки от Pinchtab

Если сервер тормозит — проверить первым делом:
```
ps aux --sort=-%cpu | grep headless-shell
```
Если есть renderer с большим `TIME` (накопленное CPU-время) — Pinchtab виновник.
Проверить открытые вкладки: `browser.sh tabs`
Остановить: `~/.pinchtab/run.sh stop`

### Pinchtab — не постоянный сервис

Запускать только когда нужен, останавливать после завершения задачи. Не оставлять работать "на всякий случай".
